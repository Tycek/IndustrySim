# Market Stockpile Redesign — Implementation Plan

## Goal

Replace the current random buy/sell offer generation with a stockpile-driven market. The market
represents external world demand. It holds a stockpile per resource (for resources it demands),
posts one persistent buy offer per resource (capacity-limited), and consumes its stockpile at a
fixed rate each turn.

Prices continue to be driven by the existing drift-based `AdjustPrices` mechanism, enhanced with
additional buy/sell pressure derived from the stockpile fill ratio for market-demanded resources.
Intermediate goods (coal, crude oil, etc.) that the market does not consume have their prices
driven purely by actual offer activity from AI companies and the player — no change to that
mechanism.

No sell offers are generated by the market. Raw materials only enter the economy through player
and AI company industries (mines).

---

## New Market State

Add to `Market.cs`:

```csharp
// Current stockpile level per resource (only for resources the market demands).
public Dictionary<string, double> StockpileLevel { get; set; }

// Maximum stockpile capacity per resource. Grows slowly each turn.
public Dictionary<string, double> StockpileCapacity { get; set; }

// Units the market consumes from its stockpile per turn (fixed baseline).
// Also acts as the pressure scale for the fill-ratio price signal.
// Only resources present in this dictionary have a market stockpile.
public static readonly IReadOnlyDictionary<string, double> ConsumptionRates
```

### Which Resources Have a Market Stockpile

Only finished/processed goods the outside world buys. Intermediate goods used only inside
the supply chain (Coal, Iron Ore, Coal Coke, Copper Ore, Crude Oil, Plastic Resin) have
**no stockpile** — their prices are driven entirely by inter-company offer activity.

Resources with a market stockpile (example consumption rates, units/turn — tune during playtesting):

| Resource         | Consumption/turn |
|------------------|-----------------|
| Steel Ingot      | 5               |
| Steel Wire       | 4               |
| Steel Bar        | 4               |
| Steel Plate      | 4               |
| Copper Ingot     | 4               |
| Copper Wire      | 3               |
| Copper Pipe      | 3               |
| Gas              | 5               |
| Diesel           | 5               |
| Kerosene         | 5               |
| Chemicals        | 4               |
| Plastics         | 5               |
| Fertiliser       | 3               |
| Synthetic Rubber | 3               |

Chemicals is dual-use (input to Plastic Plant AND bought by market). Its stockpile pressure
stacks additively with offer activity from industry chains — both are legitimate shortage signals.

Starting capacities = ConsumptionRate × 10 (gives ~10 turns of buffer at zero production;
adjust during playtesting).

### Game-Start Randomization

At `GameLoop.StartNew`, randomize the initial `StockpileLevel` for each market-demanded resource:

```
StockpileLevel[r] = StockpileCapacity[r] × uniform(0.20, 0.80)
```

This ensures no two games start with the same price landscape.

## Price Mechanism

`AdjustPrices(sellPressure, buyPressure)` **is kept** and continues to drive prices via the
existing drift logic. What changes is the input to that call: before invoking it, the stockpile
fill ratio for market-demanded resources contributes additional pressure on both sides.

### Stockpile Pressure Contribution

For each resource `r` that has a market stockpile:

```
fillRatio        = StockpileLevel[r] / StockpileCapacity[r]   // clamped [0, 1]
scale            = ConsumptionRates[r]

extraBuyPressure[r]  += (1 - fillRatio) × scale   // empty stockpile → maximum buy boost
extraSellPressure[r] += fillRatio × scale          // full stockpile  → maximum sell boost
```

These are added to the `buyPressure` and `sellPressure` dictionaries that GameLoop accumulates
from actual offer activity before passing them to `AdjustPrices`.

### Behaviour by resource type

| Resource type | Price driver |
|---|---|
| Intermediate only (Coal, Iron Ore, Crude Oil, …) | Actual AI/player offer activity only |
| Market only (Steel Wire, Copper Pipe, Diesel, …) | Stockpile pressure only (no industry demand) |
| Dual-use (Chemicals, …) | Offer activity + stockpile pressure, additive |

### Why this works for each case

- **Intermediate goods**: no stockpile, no extra pressure. Price rises only when AI companies
  actually post buy offers for a resource they're short on; price falls when they post sell
  offers they can't move. Transparent — player can see offer activity driving the price.

- **Market-only goods**: stockpile full → sell pressure spike → price falls even if no AI sell
  offers exist (nobody needs the resource, AI eventually stops producing it). Stockpile empty →
  buy pressure spike → price rises, incentivising production.

- **Dual-use goods**: both signals stack. If Chemicals are short for both plastic plants and the
  market stockpile, both pressures push price up simultaneously. If they partially cancel (e.g.
  oversupply for industries but market stockpile low), the net signal correctly reflects the
  mixed situation.

`PreviousPriceIndex` is still updated each turn for trend arrows in the UI.

## Offer Model

### One Persistent Buy Offer Per Resource

Replace the randomised multi-offer generation with one persistent `MarketOffer` per
market-demanded resource:

- `Type = OfferType.Buy`
- `Source = "Market"`
- `TurnsRemaining = int.MaxValue` (never expires naturally)
- `Quantity` = remaining free capacity, capped per turn at `ConsumptionRate[r] × 3` so players
  cannot dump unlimited stock in one turn. Rounded to nearest 5 for readability.
- `PricePerUnit = PriceIndex[r]` (refreshed each turn after prices are updated)

If `StockpileLevel[r] >= StockpileCapacity[r]`, no offer is posted for that resource.

### No Market Sell Offers

`GenerateOffers` stops generating any `OfferType.Sell` offers from `Source = "Market"`.
Sell offers may still be posted by AI companies and the player.

### Accepting a Buy Offer

When player or AI accepts a market buy offer (sells to the market):
- Deduct resource from seller inventory
- Credit seller balance
- **Increase `StockpileLevel[r]`** by the quantity sold
- Refresh the offer quantity for that resource immediately

A dedicated `market.RecordSaleToMarket(resourceName, quantity)` method handles the stockpile
update, so the same call works in `GameLoop.TryAcceptOffer` and `AiCompany.AcceptOffers`.

## Turn Tick Sequence

Each turn, replace `market.GenerateOffers(rng)` + `market.AdjustPrices(...)` in GameLoop with:

1. **Consume**: `StockpileLevel[r] -= ConsumptionRates[r]`, floor at 0 — for all market-demanded
   resources.
2. **Grow capacity**: `StockpileCapacity[r] *= (1 + GrowthRatePerTurn)`.
   Suggested `GrowthRatePerTurn = 0.002` (0.2%/turn ≈ 11% per 50 turns).
3. **Apply random events**: shift `StockpileLevel[r]` by active event modifiers, clamp to
   `[0, StockpileCapacity[r]]`, decrement event durations, remove expired events.
4. **Compute stockpile pressure**: for each market-demanded resource compute `extraBuyPressure`
   and `extraSellPressure` from fill ratio × `ConsumptionRates[r]`.
5. **Add to offer-activity pressure**: merge with `sellPressure`/`buyPressure` accumulated from
   actual offer acceptances this turn (already tracked in GameLoop).
6. **Adjust prices**: call `market.AdjustPrices(combinedSellPressure, combinedBuyPressure)`.
7. **Refresh persistent offers**: rebuild the single buy offer per resource using the updated
   `PriceIndex` and current stockpile gap.
8. **Tick AI company offers/contracts**: existing expiry logic unchanged.

## Random Events

Add to `Market.cs`:

```csharp
public List<MarketEvent> ActiveEvents { get; set; } = [];
```

```csharp
public record MarketEvent(
    string ResourceName,
    double StockpileShiftPerTurn,  // negative = drains faster (demand spike), positive = fills faster (glut)
    int TurnsRemaining,
    string Description
);
```

Events shift `StockpileLevel[r]` each turn in step 3 above. Only market-demanded resources can
have events (intermediates have no stockpile to shift).

### Event Generation

`GameLoop.ProcessTurn` rolls for a new event with low probability (e.g. 5% per turn).

| Description               | Resource         | ShiftPerTurn | Duration   |
|---------------------------|------------------|--------------|------------|
| Construction boom         | Steel Ingot      | −8           | 8–12 turns |
| Infrastructure drive      | Steel Plate      | −6           | 6–10 turns |
| Electrical grid expansion | Copper Wire      | −5           | 8–15 turns |
| Oil glut                  | Diesel / Kerosene| +10 each     | 5–8 turns  |
| Plastics surplus          | Plastics         | +10          | 4–7 turns  |
| Chemical plant shutdown   | Chemicals        | −6           | 5–10 turns |
| Fertiliser demand spike   | Fertiliser       | −5           | 6–10 turns |

Events are reported in `TurnEvents.MarketEvents` and shown in the UI notification banner.

## Changes by File

### `Market.cs`
- Add `StockpileLevel`, `StockpileCapacity`, `ConsumptionRates`, `ActiveEvents`.
- Add `RecordSaleToMarket(string resourceName, double quantity)` — increments `StockpileLevel`,
  clamps to capacity. Called whenever a market buy offer is accepted.
- Add `ComputeStockpilePressure()` — returns `(buyPressure, sellPressure)` dictionaries from
  fill ratios × consumption rates for all market-demanded resources.
- Add `RefreshPersistentOffers()` — rebuilds the one persistent buy offer per resource.
- Add `TickStockpiles(Random rng)` — handles steps 1–3 of the turn sequence above
  (consume, grow capacity, apply events). Called before price adjustment.
- Keep `AdjustPrices(sellPressure, buyPressure)` — unchanged internally; caller now passes
  combined pressures.
- Remove random sell offer generation from `GenerateOffers`; repurpose or remove the method
  (AI/player contract ticking can remain in a slimmed-down version if needed).

### `GameLoop.cs`
- Call `market.TickStockpiles(rng)` at the start of the turn sequence.
- Accumulate `sellPressure` / `buyPressure` from offer acceptances as before.
- Call `market.ComputeStockpilePressure()` and merge into accumulated pressures.
- Call `market.AdjustPrices(combinedSell, combinedBuy)`.
- Call `market.RefreshPersistentOffers()` after prices update.
- Add random event roll + selection logic.
- Propagate event descriptions into `TurnEvents`.
- Initialize stockpiles at `GameLoop.StartNew` with randomised fill levels.

### `AiCompany.cs`
- `AcceptOffers`: when accepting a market buy offer, call `market.RecordSaleToMarket(...)`.

### `GameLoop.TryAcceptOffer`
- Same `market.RecordSaleToMarket(...)` call on market buy offer acceptance.

### `TurnEvents.cs`
- Add `IReadOnlyList<string> MarketEvents` (event descriptions that fired this turn).

## AI Company Integration — Expected Margin Formula

In `AiCompany.ConsiderBuildingIndustry`, add a margin filter before the weighted pick:

```csharp
private static decimal ComputeExpectedMargin(IIndustry industry, IReadOnlyDictionary<string, decimal> priceIndex)
{
    var revenue = industry.OutputsProduced.Sum(o =>
        (decimal)o.Quantity * priceIndex.GetValueOrDefault(o.Name, 0m));
    var inputCost = industry.InputsRequired.Sum(i =>
        (decimal)i.Quantity * priceIndex.GetValueOrDefault(i.Name, 0m));
    return revenue - inputCost - industry.RunningCost;
}
```

Filter candidates to only those with `margin > 0` before `WeightedPick`. If no candidates pass,
skip building entirely — no point building an unprofitable industry at current prices.

`AiCompanyGenerator.Score()` stays as-is (supply/demand balance for *which* industry to build);
the margin formula acts as a gate (*whether* to build at all).
